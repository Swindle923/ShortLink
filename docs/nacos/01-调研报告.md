# Nacos 动态配置中心调研报告

## 1. 调研目标
- 在 Docker 环境部署 Nacos，统一管理业务配置、策略规则和动态下发内容。
- 评估主流测试平台与 Nacos 集成可行性，形成可执行实施路线。

## 2. 现有系统基线
- 业务架构：`admin + project + aggregation`（单体聚合启动）。
- 配置现状：以 `application.yaml` + `.env` 为主，运行期动态变更能力弱。
- 现有可观测：Prometheus + Grafana + Alertmanager + Loki。

## 3. 主流测试平台调研
### 3.1 Apache JMeter
- 技术架构：Java 客户端/非 GUI 执行，脚本（JMX）驱动，多线程压测模型。
- 功能特性：HTTP/DB/JDBC/消息协议覆盖广，插件生态成熟。
- 部署方式：本机、Docker、分布式 Master/Slave。
- 扩展能力：BeanShell/Groovy、插件 SDK、CI 集成（Jenkins/GitLab）。
- 与 Nacos 集成可行性：高。
  - 可在压测前通过 Nacos OpenAPI 拉取配置生成变量文件。
  - 可在不同场景用 DataId/Group 切换策略。

### 3.2 k6
- 技术架构：Go Runtime + JS 场景脚本，CLI 驱动。
- 功能特性：脚本可编程性强，适合 API 性能测试与 CI 快速回归。
- 部署方式：本机、Docker、k6-operator（K8s）。
- 扩展能力：阈值断言、趋势输出、Prometheus remote write。
- 与 Nacos 集成可行性：高。
  - 测试启动前拉取 Nacos 配置并注入环境变量。
  - 可将压测策略（QPS、VU、阈值）放在 Nacos 动态配置。

### 3.3 Locust
- 技术架构：Python 协程压测，Master/Worker 分布式。
- 功能特性：复杂业务流表达能力强，适合自定义场景。
- 部署方式：本机、Docker、K8s。
- 扩展能力：Python 生态丰富，可接入自定义数据源。
- 与 Nacos 集成可行性：高。
  - 通过 Python 客户端/HTTP API 读取 Nacos 配置，动态构造用户行为模型。

### 3.4 Postman/Newman（API 回归）
- 技术架构：集合驱动，Newman CLI 执行。
- 功能特性：接口回归与契约验证友好。
- 部署方式：本机、CI Runner、Docker。
- 扩展能力：脚本化前后置逻辑有限但够用。
- 与 Nacos 集成可行性：中高。
  - 可在执行前拉取 Nacos 配置生成环境变量文件。
  - 更适合回归验证，不是高并发压力主平台。

## 4. 平台横向对比结论
- 性能压测主平台建议：`k6`（轻量、脚本化、CI 友好）+ `JMeter`（协议覆盖和团队认知）。
- 业务流复杂场景补充：`Locust`。
- 接口回归：`Newman`。
- 与 Nacos 集成方式统一采用：`测试执行前拉取配置 + 场景运行时按需读取`。

## 5. Nacos 在本项目中的适配性
- 适配点高：
  - 现有大量配置项来自 `@ConfigurationProperties`（如风控、白名单、演示模式）。
  - 运行环境已 Docker 化，接入 Nacos 成本低。
- 主要风险：
  - 部分 `@Value` 常量类配置未统一进动态配置域。
  - 配置误发布可能影响核心链路，需要灰度发布和回滚机制。

## 6. 调研结论
- Nacos 可作为本项目统一动态配置中心，技术可行性高，实施风险可控。
- 推荐采用“核心策略先迁移、再逐步扩大覆盖”的分阶段集成方案。
