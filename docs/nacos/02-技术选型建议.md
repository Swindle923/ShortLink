# Nacos 技术选型建议

## 1. 结论摘要
- 配置中心：`Nacos 2.2.3-slim（Docker Standalone）`
- 配置接入：`spring-cloud-starter-alibaba-nacos-config`
- 动态刷新方式：`@RefreshScope + @ConfigurationProperties`
- 发布策略：`共享配置 + 服务私有配置 + 分组隔离`

## 2. 选型理由
### 2.1 为什么选 Nacos
- 与 Spring Cloud Alibaba 体系天然兼容。
- 配置管理、分组、命名空间、历史版本能力完整。
- 支持长轮询推送，适合策略类配置实时下发。
- Docker 部署轻量，适合毕业设计与中小规模项目。

### 2.2 为什么选 2.2.3-slim
- 在 Apple Silicon（M 系列）环境兼容性更稳。
- 兼容当前项目依赖的 `nacos-client-2.2.1` 生态。
- 资源占用较低，演示环境启动快。

## 3. 配置模型建议
### 3.1 DataId 设计
- `shortlink-common.yaml`：全局策略（流控、白名单）
- `shortlink.yaml`：服务级配置（演示模式、CORS）

### 3.2 Group/Namespace 设计
- Group：`SHORTLINK_GROUP`（同环境同业务域）
- Namespace：
  - 本地可留空
  - 测试/预发/生产建议独立 Namespace

### 3.3 Key 规范
- 统一前缀 `short-link.*`
- 策略类按领域分层：
  - `short-link.flow-limit.*`
  - `short-link.goto-domain.white-list.*`
  - `short-link.demo-mode.*`

## 4. 测试平台选型建议
- 压测主平台：k6（首选）+ JMeter（补充）
- 回归平台：Newman
- 复杂业务流：Locust

并统一接入 Nacos 配置读取能力，保证测试环境与业务策略同步。

## 5. 不建议做法
- 不建议一开始把全部配置直接迁入 Nacos（风险高，排障复杂）。
- 不建议用单 DataId 承载所有配置（变更冲击面太大）。
- 不建议无版本灰度直接覆盖生产配置。
