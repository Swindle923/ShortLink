# Nacos 集成实施方案（详细版）

## 0. 交付范围
本方案覆盖：
1) Nacos 在 Docker 环境中的部署配置  
2) 业务系统与 Nacos 的集成架构设计  
3) 配置下发流程和策略管理机制  
4) 动态配置更新的可靠性保障  
5) 集成后的测试验证方案

---

## 1) Docker 部署配置方案

### 1.1 部署文件
- `docker-compose.nacos.yml`
- `scripts/start-nacos.sh`
- `scripts/stop-nacos.sh`

### 1.2 部署参数
- 端口：`8848/9848/9849`
- 模式：`standalone`
- 鉴权：开启（用户名密码 + token）
- 数据卷：`nacos_data`、`nacos_logs`

### 1.3 启动流程
```bash
bash ./scripts/start-nacos.sh
```

### 1.4 基础配置初始化
- 脚本：`scripts/nacos-publish-config.sh`
- 发布 DataId：
  - `shortlink-common.yaml`
  - `short-link-aggregation.yaml`

---

## 2) 业务系统与 Nacos 集成架构设计

### 2.1 应用侧改造
- 依赖：
  - `aggregation/pom.xml` 增加 `spring-cloud-starter-alibaba-nacos-config`
  - `admin/pom.xml` / `project/pom.xml` 增加 `spring-cloud-context`
- 配置入口：
  - `aggregation/src/main/resources/application-nacos.yaml`
  - 通过 `spring.config.import` 引入 Nacos DataId
- 启动脚本：
  - `scripts/start-monolith-nacos.sh`

### 2.2 架构分层
- 管理面：Nacos Console 统一配置管理
- 配置接入层：Spring Cloud Nacos Config
- 业务消费层：
  - `DemoModeProperties`
  - `UserFlowRiskControlConfiguration`
  - `GotoDomainWhiteListConfiguration`

### 2.3 动态刷新对象
- 已加 `@RefreshScope` 的配置类支持热更新：
  - `aggregation/config/DemoModeProperties`
  - `admin/config/UserFlowRiskControlConfiguration`
  - `project/config/GotoDomainWhiteListConfiguration`

## 3) 配置下发流程与策略管理机制

### 3.1 配置分层策略
- 共享策略（shared）：
  - `shortlink-common.yaml`
  - 放置全局治理策略，如流控、白名单
- 服务私有（service）：
  - `short-link-aggregation.yaml`
  - 放置聚合服务特定配置，如 demo-mode、CORS

### 3.2 下发流程
1. 运维/研发在 Nacos Console 修改配置。
2. Nacos 推送变更通知到客户端。
3. Spring Cloud 刷新 `@RefreshScope` Bean。
4. 新请求读取新配置生效（不中断服务）。

### 3.3 策略管理机制
- 命名约定：`short-link.<domain>.<item>`
- 变更规范：
  - 先测试 Group/Namespace 验证
  - 再灰度到目标环境
- 回滚方式：
  - 使用 Nacos 历史版本回滚
  - 或脚本重新发布上一个稳定版本

## 4) 动态配置更新可靠性保障方案

### 4.1 兜底策略
- `application.yaml` 保留默认值，Nacos 异常时可退回本地配置。
- `spring.config.import` 使用 `optional:nacos:` 避免因配置中心不可用导致硬失败（按环境策略启用）。

### 4.2 发布安全策略
- 高风险配置（风控阈值、白名单）走双人复核。
- 使用分环境 Group/Namespace，禁止跨环境误发布。
- 对策略值做边界约束（如最大最小阈值）。

### 4.3 运行期保护
- 关键链路保留默认保护值。
- 发布后通过指标看板观察：
  - 错误率、P95、配额耗尽比例
  - 风控拦截率变化

### 4.4 故障处置
- Nacos 不可用：
  - 已加载配置继续生效
  - 新发布暂停，按本地配置运行
- 配置误发布：
  - 立即回滚 DataId 历史版本
  - 触发变更告警并记录审计

## 5) 集成后测试验证方案

### 5.1 验证目标
- Nacos 可用性
- 配置发布与读取正确性
- 动态刷新生效
- 异常回退能力

### 5.2 测试用例
1. 启动验证：
   - 启动 Nacos 容器，检查健康接口
2. 发布验证：
   - 发布 `shortlink-common.yaml`，读取接口校验内容
3. 应用接入验证：
   - 以 `nacos` profile 启动聚合服务
   - 检查应用日志中的 Nacos 配置加载信息
4. 动态刷新验证：
   - 修改 `flow-limit.max-access-count`
   - 观察服务行为/指标变化是否符合新阈值
5. 异常验证：
   - 暂停 Nacos，确认业务服务不崩溃且保留已加载配置

### 5.3 验证产物
- 命令执行日志
- 关键接口响应
- Prometheus/Grafana 指标对比图
- 变更前后行为对照表
