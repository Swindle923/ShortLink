# 07 六大模块原理实现（代码级）

> 本文对应前 6 份功能文档，重点说明“在代码中是怎么做出来的”。

## 1. 短链接生成原理（对应文档 01）
### 1.1 分层调用
- 管理端入口：`admin/controller/ShortLinkController#createShortLink`
- 统一能力接口：`admin/remote/ShortLinkActualRemoteService`
- 单体本地实现：`aggregation/remote/LocalShortLinkActualRemoteService#createShortLink`
- 核心业务：`project/service/impl/ShortLinkServiceImpl#createShortLink`

### 1.2 核心实现步骤
1. 参数映射：`admin DTO -> project DTO`（`LocalShortLinkActualRemoteService` 内通过 JSON 转换）。
2. 生成短码：在 `ShortLinkServiceImpl` 里构建 `shortUri/fullShortUrl`。
3. 写入主表与跳转映射表：
   - `t_link`
   - `t_link_goto`
4. 冲突处理：捕获 `DuplicateKeyException`，走重试/报错分支。
5. 布隆过滤器更新：减少后续不存在短链的穿透查询。

### 1.3 关键设计
- 索引 + 异常兜底保证唯一性。
- 支持分布式锁版本创建（`create/by-lock`）应对高并发冲突。
- 生成阶段同步写入 `max_access_count/current_access_count`，为配额跳转做基础。

## 2. 短链接跳转原理（对应文档 02）
### 2.1 入口与核心方法
- 跳转逻辑核心：`project/service/impl/ShortLinkServiceImpl#restoreUrl`
- 配额消费核心：`ShortLinkServiceImpl#tryConsumeAccessQuota`

### 2.2 处理流程
1. 先查缓存（正向缓存 + 空值缓存）。
2. 缓存未命中再查库（短链映射 -> 主表详情）。
3. 校验状态（启用、过期、配额）。
4. 配额扣减：
   - 有上限：原子 `current_access_count + 1`，超限自动禁用短链并回写空值缓存。
   - 无上限：直接放行。
5. 重定向前后异步投递统计消息。

### 2.3 关键设计
- 缓存优先，降低 DB 压力。
- 空值缓存 + 布隆过滤器，防穿透。
- 跳转路径与统计路径解耦，保证用户体验。

## 3. 追踪统计原理（对应文档 03）
### 3.1 消息链路
- 生产者：`project/mq/producer/ShortLinkStatsSaveProducer#send`
- 消费者：`project/mq/consumer/ShortLinkStatsSaveConsumer#onMessage`

### 3.2 统计落地
消费后调用 `actualSaveShortLinkStats`，写入多张统计表：
- `t_link_access_stats`
- `t_link_locale_stats`
- `t_link_os_stats`
- `t_link_browser_stats`
- `t_link_device_stats`
- `t_link_network_stats`
- `t_link_access_logs`
- `t_link_stats_today`

并回写 `t_link` 总量计数（PV/UV/UIP）。

### 3.3 一致性与可靠性
- 幂等控制：`MessageQueueIdempotentHandler`，避免重复消费放大统计。
- 锁控制：分组更新与统计并发时，用读写锁保持一致。
- 失败可重试：消费异常回滚幂等标记。

## 4. 分组管理原理（对应文档 04）
### 4.1 接口与服务
- 控制器：`admin/controller/GroupController`
- 服务：`admin/service/impl/GroupServiceImpl`

### 4.2 能力实现
- 新增/列表/修改/删除/排序分组均由 `GroupServiceImpl` 实现。
- 用户注册成功后自动创建默认分组（在 `UserServiceImpl#register` 中调用 `groupService.saveGroup(...)`）。
- 短链分页、回收站、统计查询都以 `gid` 或 `gidList` 作为过滤边界。

### 4.3 关键设计
- 分组是“查询边界 + 管理边界”，不是简单标签。
- 前后端围绕 `gid` 一致建模，减少后续扩展复杂度。

## 5. 可观测性与日志原理（对应文档 05）
### 5.1 指标链路
- 应用导出：Actuator + Micrometer（`/actuator/prometheus`）。
- Prometheus 抓取：`monitoring/prometheus/prometheus.yml`
- Grafana 展示：`monitoring/grafana/dashboards/*.json`
- 告警规则：`monitoring/prometheus/alert.rules.yml`
- 告警路由：`monitoring/alertmanager/alertmanager.yml`

### 5.2 日志链路
- Loki：日志存储与检索。
- Promtail：采集 Docker 容器日志（`monitoring/promtail/promtail-config.yml`）。
- Grafana Explore：统一 LogQL 查询。

### 5.3 监控工具抽象（代码级亮点）
- 自动埋点：`aggregation/config/metrics/ServiceMetricsAspect`
  - 对 `admin/project` 的 `service.impl` 自动采集调用次数、成功失败、耗时分布。
- 手工埋点工具：`admin/project/common/monitoring/BizMetricsKit`
  - 业务代码可一行上报自定义指标。

## 6. 其他核心能力原理（对应文档 06）
### 6.1 回收站
- 服务实现：`project/service/impl/RecycleBinServiceImpl`
- 能力：移入、恢复、彻底删除，结合缓存失效保证跳转行为一致。

### 6.2 RBAC 最小模型
- 用户角色字段：`t_user.role`
- 角色校验过滤器：`admin/common/biz/user/UserAuthenticationFilter`
- 角色管理接口：`/user/role`（控制器与服务已接入）

### 6.3 前端联动
- 创建/编辑页支持 `maxAccessCount`。
- 列表页展示“剩余访问次数”。
- 回收站与个人中心（角色管理）具备完整可操作页面。

## 7. 总结（实现方法论）
- 业务分层：`admin`（管理面）-> `aggregation`（编排面）-> `project`（域能力）。
- 数据一致性：索引约束 + 锁 + 幂等 + 缓存失效联动。
- 性能策略：缓存优先、统计异步、热点路径最短化。
- 可运维性：指标、日志、告警一体化，问题可观测、可定位、可回放。
