# 短链接分组管理原理（论文叙事版）

在短链接系统中，分组从来不是“前端列表上的一个分类标签”，它实际上是贯穿创建、查询、统计与权限边界的核心隔离键。当前项目把这部分逻辑集中在 `GroupServiceImpl`，控制器层的职责非常薄，主要负责请求接入和参数转发，真正的约束策略都在服务层完成。这样的设计使分组从一开始就被建模为领域规则，而不是 UI 附属功能。

分组创建流程最有代表性的实现是 `saveGroup(username, groupName)`。这段代码首先按用户名加分布式锁 `LOCK_GROUP_CREATE_KEY`，其目标不是全局串行化，而是避免“同一用户并发创建分组”时出现计数越界和 gid 冲突。锁内会先读取该用户现有分组数，并与配置项 `short-link.group.max-num` 比较。这个配额校验体现了系统治理能力：它把资源限制提前到写入前，而不是在后续查询时被动承受过量数据。

通过数量校验后，系统进入 gid 生成阶段。这里采用的是“随机生成 + Bloom 预判 + 唯一表兜底 + 重试”的组合策略。具体来说，服务先调用 `RandomGenerator.generateRandom()` 得到候选 gid，然后用 `gidRegisterCachePenetrationBloomFilter` 做存在性预判，最后真正写入 `t_group_unique`。如果出现唯一键冲突，说明候选值已被占用，流程会继续重试，最多十次。这个流程的本质是把高频判重从数据库前移到内存概率结构，同时保留数据库作为最终裁决者，从而在性能和正确性之间取得平衡。

在持久化模型上，`t_group_unique` 与 `t_group` 的双表并存有明确分工：前者只负责全局 gid 唯一性，后者负责业务属性如分组名称、排序、归属用户和删除标记。这样拆分后，唯一性冲突不会与业务字段更新耦合，也让后续软删除、重命名、排序调整等操作保持简洁。特别是在逻辑删除场景下，系统并不物理删除分组，而是通过 `delFlag` 标记状态，这为审计、恢复和历史追踪保留了空间。

分组查询也不是简单的单表返回。`listGroup` 会先按当前用户拉取分组元数据，再通过 `ShortLinkActualRemoteService.listGroupShortLinkCount(...)` 聚合每个分组的短链数量，最终在内存中按 gid 合并成响应 DTO。也就是说，管理台看到的“分组列表”是一个组合视图：既包含主数据，也包含统计衍生信息。这种做法增加了后端组装复杂度，但换来了前端体验的一致性和接口语义完整性。

更新、删除与排序这三类操作在代码上都遵循同一个安全边界：必须以 `username + gid + delFlag=0` 作为条件执行。这种条件更新的意义在于把用户隔离和数据状态判断收敛到数据库层，防止越权修改或脏状态覆盖。尤其是排序操作，虽然表面是轻量写入，但本质仍属于用户私有空间下的数据重排，因此同样必须走归属校验。

从全局架构看，分组是整个系统的“业务坐标系”。短链创建必须指定 gid，分页查询以 gid 过滤，统计查询通过 gid 限定范围，回收站分页按 gidList 聚合。也就是说，分组并不是局部模块，而是跨模块的数据组织基准。把它设计好，系统就有清晰的多租户边界；把它设计坏，后续权限、统计和治理都会陷入混乱。当前实现选择了偏稳健的路线：限制规模、强化唯一、保留软删、聚合视图，这些特征共同保证了分组体系在业务增长后仍具备可维护性。

如果要看到这套分组机制是如何“真正落地”的，最关键的是 `saveGroup` 源码。下面的代码片段直观体现了三个约束动作：按用户加锁、数量上限校验、gid 冲突重试与落库。

```java
@Override
public void saveGroup(String username, String groupName) {
    RLock lock = redissonClient.getLock(String.format(LOCK_GROUP_CREATE_KEY, username));
    lock.lock();
    try {
        List<GroupDO> groupDOList = baseMapper.selectList(
            Wrappers.lambdaQuery(GroupDO.class)
                .eq(GroupDO::getUsername, username)
                .eq(GroupDO::getDelFlag, 0)
        );
        if (CollUtil.isNotEmpty(groupDOList) && groupDOList.size() == groupMaxNum) {
            throw new ClientException(String.format("已超出最大分组数：%d", groupMaxNum));
        }
        int retryCount = 0;
        String gid = null;
        while (retryCount < 10) {
            gid = saveGroupUniqueReturnGid();
            if (StrUtil.isNotEmpty(gid)) {
                baseMapper.insert(GroupDO.builder()
                        .gid(gid).sortOrder(0).username(username).name(groupName).build());
                gidRegisterCachePenetrationBloomFilter.add(gid);
                break;
            }
            retryCount++;
        }
        if (StrUtil.isEmpty(gid)) {
            throw new ServiceException("生成分组标识频繁");
        }
    } finally {
        lock.unlock();
    }
}
```

gid 唯一性之所以稳定，不只是因为代码里做了重试，而是数据库专门有唯一表做硬约束。`saveGroupUniqueReturnGid` 最终写的就是 `t_group_unique`，一旦冲突直接抛 `DuplicateKeyException` 回到重试逻辑。

```java
private String saveGroupUniqueReturnGid() {
    String gid = RandomGenerator.generateRandom();
    if (gidRegisterCachePenetrationBloomFilter.contains(gid)) {
        return null;
    }
    try {
        groupUniqueMapper.insert(GroupUniqueDO.builder().gid(gid).build());
    } catch (DuplicateKeyException e) {
        return null;
    }
    return gid;
}
```

对应 DDL 也能看出这个设计意图，`t_group_unique` 仅做 gid 去重，不承担业务字段。

```sql
CREATE TABLE `t_group_unique` (
  `id`  bigint(20) NOT NULL AUTO_INCREMENT,
  `gid` varchar(32) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_unique_gid` (`gid`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
