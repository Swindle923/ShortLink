# 可观测性与日志监控原理（论文叙事版）

短链接系统的可观测性建设，本质上是在回答一个工程问题：当系统“看起来还能跑”但业务指标已经异常时，开发者能否在最短时间内定位问题根因。当前项目并没有把可观测性当作后期附加能力，而是把它嵌入到了应用层、服务层和基础设施层三个层次。应用层通过 Micrometer 暴露标准指标，服务层通过 AOP 与业务埋点补充语义指标，基础设施层通过 Prometheus、Alertmanager、Grafana 与 Loki 形成监控和日志闭环。

首先看指标主链路。应用启动后会通过 Actuator 暴露 Prometheus 采集端点，Prometheus 按 `monitoring/prometheus/prometheus.yml` 周期抓取样本。抓回来的时序数据一部分进入 Grafana 做可视化看板，另一部分由规则引擎按 `alert.rules.yml` 进行表达式评估，触发后再经 Alertmanager 分发告警。这条链路的关键点在于，告警不是由 Grafana 面板阈值临时判断，而是由 Prometheus 规则持续计算，因此其行为可版本化、可审计、可复现。

如果只依赖通用 JVM 与 HTTP 指标，系统很难回答“到底是哪条业务路径在恶化”。为此项目在代码层增加了两类补强机制。第一类是 `ServiceMetricsAspect`，它拦截 `admin` 与 `project` 服务实现层方法，自动产出调用计数和耗时分布，并附带模块、类名、方法名、执行结果和异常类型标签。这使方法级别的性能画像在无侵入条件下成立。第二类是 `BizMetricsKit`，它允许业务代码在关键状态点显式上报语义指标，例如短链创建成功率、冲突率、配额耗尽次数、统计查询空结果占比等。相比纯技术指标，这些语义指标更接近业务真实健康度。

日志体系由 Promtail 与 Loki 组成。Promtail 通过容器发现读取日志源，对 Docker 日志结构做 pipeline 解析并附加标签，再推送到 Loki。Grafana 的 Explore 页面使用 LogQL 查询日志流，开发者可以按服务、级别、实例、时间窗口快速筛选异常上下文。与传统“本地 grep 日志”相比，这种做法的价值在于跨实例统一检索和时间维度关联，尤其适合排查偶发错误与链路抖动。

可观测性真正的挑战不在“采集到数据”，而在“能否建立可行动的诊断路径”。当前项目在这方面已经形成了基础能力：当用户反馈跳转失败率上升时，可以先看业务指标中 `shortlink.biz.shortlink.quota.consume` 的 exhausted 比例，再关联服务层错误计数，最后进入日志系统检查具体异常堆栈；当创建接口延迟升高时，可以直接观察 `shortlink.biz.service.latency` 在 `ShortLinkServiceImpl` 相关方法的分位数变化，并与数据库或 Redis 侧指标做时间对齐。这种从业务到技术的多层联动，是监控系统“可用”与“不可用”的分水岭。

从架构适配角度看，这套方案也符合当前项目形态。你们是单体聚合部署但依赖多个基础组件，既需要轻量接入，又需要具备后续扩展空间。Prometheus + Grafana + Loki 的组合在复杂度和收益之间处于较好的平衡点：部署成本可控，查询能力完整，且不会把监控能力绑定到特定云厂商。随着系统规模增长，可以在现有体系上继续补充链路追踪、告警分级和自动化 Runbook，而不需要推倒重来。

因此，当前可观测性设计的真正贡献不只是“看板更好看”，而是把系统运行状态转化为可量化、可解释、可处置的证据链。对于毕业设计来说，这部分内容能够很好体现你对工程实践的理解深度，因为它证明你不仅关注功能实现，也关注系统在真实运行环境中的可维护性与可运营性。

如果把这套能力具体到代码，`ServiceMetricsAspect` 是最关键的自动化埋点入口。它通过一个切面同时拦截 `admin.service.impl` 和 `project.service.impl`，把方法级调用次数和耗时指标自动上报到 Micrometer。也就是说，开发者不需要在每个服务方法手写计时逻辑，系统就能天然得到成功率和延迟分布。

```java
@Around("execution(* com.nageoffer.shortlink.admin.service.impl..*(..)) || execution(* com.nageoffer.shortlink.project.service.impl..*(..))")
public Object aroundServiceCall(ProceedingJoinPoint joinPoint) throws Throwable {
    String className = joinPoint.getSignature().getDeclaringTypeName();
    String methodName = joinPoint.getSignature().getName();
    String simpleClassName = joinPoint.getSignature().getDeclaringType().getSimpleName();
    String module = className.contains(".admin.") ? "admin" : "project";
    long startNanos = System.nanoTime();
    try {
        Object result = joinPoint.proceed();
        Counter.builder("shortlink.biz.service.calls")
                .tag("module", module).tag("class", simpleClassName)
                .tag("method", methodName).tag("result", "success")
                .register(meterRegistry).increment();
        return result;
    } catch (Throwable ex) {
        Counter.builder("shortlink.biz.service.calls")
                .tag("module", module).tag("class", simpleClassName)
                .tag("method", methodName).tag("result", "error")
                .tag("exception", ex.getClass().getSimpleName())
                .register(meterRegistry).increment();
        throw ex;
    } finally {
        Timer.builder("shortlink.biz.service.latency")
                .publishPercentileHistogram(true)
                .tag("module", module).tag("class", simpleClassName).tag("method", methodName)
                .register(meterRegistry)
                .record(System.nanoTime() - startNanos, TimeUnit.NANOSECONDS);
    }
}
```

除了自动埋点，项目还提供了 `BizMetricsKit` 作为业务语义指标的统一入口。它看起来只是一个小工具类，但价值在于把指标命名、标签规范和 Micrometer 调用方式收敛成统一 API，防止后期出现“同一业务多个指标名”的治理混乱。

```java
public void increment(String metricName, String... tags) {
    Counter.builder(metricName)
            .tags(normalizeTags(tags))
            .register(meterRegistry)
            .increment();
}

public void recordLatency(String metricName, long nanos, String... tags) {
    Timer.builder(metricName)
            .publishPercentileHistogram(true)
            .tags(normalizeTags(tags))
            .register(meterRegistry)
            .record(nanos, TimeUnit.NANOSECONDS);
}
```

从业务调用处也能看到这个抽象是如何使用的，比如创建短链成功、冲突、配额耗尽、用户登录复用 token 等场景都在服务层显式打点，形成了“技术指标 + 业务指标”并行观测体系。这一点在故障排查中非常关键，因为它让你不仅知道“系统慢了”，还知道“是哪个业务状态导致慢了”。
