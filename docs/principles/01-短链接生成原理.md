# 短链接生成原理（论文叙事版）

短链接创建看似只是一个“把长链接缩短”的动作，但从系统实现角度，它其实是一个同时受吞吐、冲突概率、一致性和安全约束的综合问题。当前项目在代码上把这条链路放在 `ShortLinkServiceImpl#createShortLink` 与 `#createShortLinkByLock` 两个方法中，前者是默认高性能路径，后者是冲突场景下的保守路径。二者共享同一套数据模型和业务语义，只是在并发控制策略上不同，这种设计使系统既有高吞吐常态，也有可切换的稳态兜底。

从调用关系看，创建请求会先经过管理侧控制器，再通过 `ShortLinkActualRemoteService` 进入聚合层的本地实现，最终落到 `project` 域服务。这个分层不是形式上的“多写几层”，而是把协议层和领域层边界显式化。换句话说，`admin` 负责“怎么和前端交互”，`project` 负责“短链业务如何成立”，中间通过抽象接口连接。这样做的直接收益是，后续即使从单体迁移到远程调用，核心生成逻辑几乎不需要重写。

真正进入创建逻辑后，第一步并不是马上算短码，而是做目标地址白名单校验。`verificationWhitelist` 的意义在于把安全约束前移到入口，尤其在演示环境中可以防止把平台当作开放重定向中转站滥用。通过这一步过滤，系统把“可创建链接”从全互联网收敛到受信域名集合，后续的哈希生成和写库都建立在这个安全前提上。

短码生成本身采用 `MurmurHash32 + Base62`，并不是雪花算法。代码上会把 `originUrl` 与随机 UUID 拼接后再哈希，这个细节非常关键，因为它意味着同一个原始链接重复创建时不会被强制映射成同一个短码，从而降低热点同源输入导致的碰撞风险。MurmurHash 的优势在于速度快、分布较均衡，Base62 的优势在于字符集短小且 URL 友好，两者组合更像一个偏工程效率的选择，而不是追求数学完美唯一的编码体系。

但是任何哈希体系都不可能靠概率结构保证绝对唯一，因此系统没有把唯一性交给算法，而是交给了“布隆过滤器预判 + 数据库唯一键裁决”的双层机制。`shortUriCreateCachePenetrationBloomFilter` 只负责快速拒绝明显冲突候选，降低数据库压力；一旦并发竞争或误判发生，最终仍由数据库唯一约束兜底，`DuplicateKeyException` 会被明确捕获并转换为业务异常。这个边界非常清晰：Bloom 管性能，唯一键管正确性。

在数据写入上，创建动作不是单表插入，而是必须同时写 `t_link` 和 `t_link_goto`。前者承载完整业务状态，包括有效期、启用状态、统计累计值、访问配额和软删除语义；后者承载短链到分组的轻量路由关系。这样拆分的根本原因是跳转链路需要高效定位 `gid`，而统计、回收站、配额等又依赖主表完整字段。如果只保留一张表，要么跳转查得慢，要么表结构被迫承担过多冲突职责。

事务边界由 `@Transactional` 保证，这使得双表写入具有原子性：要么都成功，要么都回滚。更值得注意的是缓存预热的位置。系统选择在数据库写成功后再写 `short-link:goto:%s` 缓存，并按有效期计算 TTL，避免“缓存先写成功但事务回滚”的幽灵短链问题。这个顺序安排体现了“数据库是真相源，缓存是派生状态”的一致性原则。

`create/by-lock` 路径则展示了系统的第二种并发哲学。在默认无锁路径里，系统更依赖概率结构和数据库约束，吞吐高但在极端冲突下会有更多重试；在加锁路径里，系统通过 `SHORT_LINK_CREATE_LOCK_KEY` 把生成窗口串行化，冲突率会明显下降，但吞吐上限也被锁住。两条路径并存本质上是把“性能优先”与“稳定优先”做成了可切换策略，而不是让单一方案承担全部场景。

如果从论文视角给这条链路做总结，当前实现最有价值的不是某个单点技术，而是边界划分清楚：安全校验负责入口准入，哈希算法负责候选生成，Bloom 负责性能预筛，唯一键负责最终裁决，事务负责双表原子性，缓存负责读优化且遵循后置写入原则。正因为每一层都知道自己解决什么问题，系统在面对高并发创建、偶发冲突和异常回滚时才保持了可解释性和可演进性。

为了把实现细节落到“代码可执行语义”层面，下面给出创建链路中的关键片段。首先是 `createShortLink` 的主流程，它清楚地展示了“先校验、再生成、后落库、最后预热缓存”的时序，并且在冲突异常时打点和抛出业务异常。

```java
@Transactional(rollbackFor = Exception.class)
@Override
public ShortLinkCreateRespDTO createShortLink(ShortLinkCreateReqDTO requestParam) {
    verificationWhitelist(requestParam.getOriginUrl());
    String shortLinkSuffix = generateSuffix(requestParam);
    String fullShortUrl = StrBuilder.create(createShortLinkDefaultDomain)
            .append("/")
            .append(shortLinkSuffix)
            .toString();
    ShortLinkDO shortLinkDO = ShortLinkDO.builder()
            .domain(createShortLinkDefaultDomain)
            .originUrl(requestParam.getOriginUrl())
            .gid(requestParam.getGid())
            .validDateType(requestParam.getValidDateType())
            .validDate(requestParam.getValidDate())
            .shortUri(shortLinkSuffix)
            .maxAccessCount(normalizeMaxAccessCount(requestParam.getMaxAccessCount()))
            .fullShortUrl(fullShortUrl)
            .favicon(getFavicon(requestParam.getOriginUrl()))
            .build();
    ShortLinkGotoDO linkGotoDO = ShortLinkGotoDO.builder()
            .fullShortUrl(fullShortUrl)
            .gid(requestParam.getGid())
            .build();
    try {
        baseMapper.insert(shortLinkDO);
        shortLinkGotoMapper.insert(linkGotoDO);
    } catch (DuplicateKeyException ex) {
        if (!shortUriCreateCachePenetrationBloomFilter.contains(fullShortUrl)) {
            shortUriCreateCachePenetrationBloomFilter.add(fullShortUrl);
        }
        bizMetricsKit.increment("shortlink.biz.shortlink.create", "result", "duplicate", "mode", "single");
        throw new ServiceException(String.format("短链接：%s 生成重复", fullShortUrl));
    }
    stringRedisTemplate.opsForValue().set(
            String.format(GOTO_SHORT_LINK_KEY, fullShortUrl),
            requestParam.getOriginUrl(),
            LinkUtil.getLinkCacheValidTime(requestParam.getValidDate()), TimeUnit.MILLISECONDS
    );
    shortUriCreateCachePenetrationBloomFilter.add(fullShortUrl);
    return ShortLinkCreateRespDTO.builder().fullShortUrl("http://" + shortLinkDO.getFullShortUrl()).build();
}
```

接下来是短码候选生成逻辑。注意这里没有使用数据库自增 ID，也没有雪花算法，而是 `originUrl + UUID` 经过哈希编码后的字符串。循环重试上限是 10 次，超过即认为当前冲突压力异常。

```java
private String generateSuffix(ShortLinkCreateReqDTO requestParam) {
    int customGenerateCount = 0;
    String shorUri;
    while (true) {
        if (customGenerateCount > 10) {
            throw new ServiceException("短链接频繁生成，请稍后再试");
        }
        String originUrl = requestParam.getOriginUrl();
        originUrl += UUID.randomUUID().toString();
        shorUri = HashUtil.hashToBase62(originUrl);
        if (!shortUriCreateCachePenetrationBloomFilter.contains(createShortLinkDefaultDomain + "/" + shorUri)) {
            break;
        }
        customGenerateCount++;
    }
    return shorUri;
}
```

最后是数据库层最终一致性兜底所在。`t_link` 的唯一索引不是单列 `full_short_url`，而是 `(full_short_url, del_time)`，这让系统在软删除历史保留与新记录重建之间保持兼容。

```sql
CREATE TABLE `t_link` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `full_short_url` varchar(128) DEFAULT NULL,
  `del_time` bigint(20) DEFAULT '0',
  `del_flag` tinyint(1) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_unique_full-short-url` (`full_short_url`, `del_time`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
