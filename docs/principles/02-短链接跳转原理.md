# 短链接跳转原理（论文叙事版）

短链接系统真正的价值，不在于“能生成短码”，而在于“访问瞬间的稳定跳转能力”。在代码实现里，这条关键路径由 `ShortLinkController#restoreUrl` 进入，核心逻辑集中在 `ShortLinkServiceImpl#restoreUrl`。从请求到响应的全过程可以理解为一次对成本极其敏感的决策链：系统不断尝试用更便宜的判断提前结束请求，只有在必须时才进入昂贵路径。也正是这种“层层短路”的结构，使它在高并发场景下仍能保持可接受的延迟。

进入方法后，系统首先做的是短链标识归一化，它根据 `serverName` 与 `serverPort` 重新拼装 `fullShortUrl`，确保同一短链在不同访问上下文中映射到统一键空间。随后立即读取 Redis 正向缓存 `short-link:goto:%s`。如果命中，系统不会再做数据库查询，而是直接进入访问配额校验与重定向。这一步体现了跳转系统的根本哲学：缓存命中就是成功路径，数据库不应成为主车道。

如果正向缓存未命中，系统并不会立刻回源，而是继续做“廉价判空”。它先检查空值缓存 `short-link:is-null:goto_%s`，如果该键存在，说明此前已经确认该短链不可用、已失效或不存在，本次请求可以直接终止。若空值缓存也未命中，再通过布隆过滤器做存在性预判。Bloom 明确不包含时，直接判定无效请求；Bloom 可能包含时，才有资格进入下一阶段。这一组合对抗的是典型缓存穿透流量，它把大量垃圾请求拦在了数据库之前。

真正的回源发生在锁保护区内。系统按短链粒度获取 `LOCK_GOTO_SHORT_LINK_KEY`，目的不是“让并发变慢”，而是避免缓存击穿时多个线程同时压向数据库。在锁内，代码再次做正向缓存与空值缓存检查，这个二次检查非常关键，因为等待锁的线程很可能在此时已经能够复用其他线程刚刚回填的结果。若仍需回源，系统先查 `t_link_goto` 获取 `gid`，再以 `gid + full_short_url + 状态条件` 查询 `t_link`，同时校验有效期。至此，短链是否可跳转才算被最终裁决。

裁决通过后并不意味着可以立即跳转，系统还要做一次并发语义很强的访问配额扣减。`tryConsumeAccessQuota` 最核心的设计在于它没有走“先查剩余额度，再更新计数”这种易竞态流程，而是使用单条条件更新 SQL：只有当 `current_access_count < max_access_count` 时才允许自增。这让“校验与扣减”在数据库原子完成，避免多线程同时超限通过。若影响行数为 0，系统会把该短链置为禁用，并删除正向缓存、写入空值缓存，使后续请求快速失败并稳定收敛。

跳转路径中还有一个常被误解的点：统计采集并不会阻塞重定向。系统会在当前请求中构建统计记录对象，计算 UV 与 UIP 的“是否首次”标志，然后把消息写入 Redis Stream，随后立即执行 302。也就是说，跳转成功和统计落库不是同一个事务，而是异步最终一致关系。对用户体验而言，这个拆分收益非常直观；对系统可靠性而言，它把复杂度转移到了后续消费幂等与补偿机制中。

UV 与 UIP 的口径也值得特别说明。UV 依赖 `uv` Cookie 与 Redis Set 判重，UIP 依赖真实客户端 IP 与 Redis Set 判重，二者都在访问当下形成“首次标记”，并随消息传递到消费端。这个方案并不追求“全站用户身份学意义上的绝对唯一”，而是面向短链运营分析的局部口径一致性。换句话说，它是业务统计模型，不是账号系统模型。

如果从失败行为看这条链路，它并不是简单返回 404，而是尽量把失败结果缓存化。不存在、过期、禁用、配额耗尽这些状态都会通过空值缓存沉淀一段时间，减少重复回源成本。这种策略提高了系统在异常流量下的韧性，但也引入了短暂负缓存窗口，意味着某些状态修复后需要缓存失效或主动清理才能立即生效。当前实现已经在更新、恢复等管理动作中做了相关缓存清理，尽量降低这种窗口影响。

总体而言，这套跳转实现的技术价值并不在某个“高级组件”，而在一条经过精心排序的执行链：先缓存命中，再空值判定，再 Bloom 预判，再锁内双检，再数据库裁决，再原子配额扣减，最后异步统计与重定向。它把高并发读路径中最常见的风险点逐层拆解，并把每一层都设计成可观察、可恢复、可解释的工程单元。这正是一个成熟短链接系统在真实流量下能够长期运行的根本原因。

配额扣减正确性的关键不在 Java 代码，而在 Mapper SQL。这里的 `incrementCurrentAccessCount` 把状态校验和计数递增放在同一条语句内，避免并发下的“先查后改”竞态。随后 `disableByAccessLimit` 在超限后执行状态收敛。

```sql
<update id="incrementCurrentAccessCount">
    UPDATE t_link
    SET current_access_count = current_access_count + 1
    WHERE gid = #{gid}
      AND full_short_url = #{fullShortUrl}
      AND del_flag = 0
      AND enable_status = 0
      AND (max_access_count IS NULL OR current_access_count < max_access_count)
</update>

<update id="disableByAccessLimit">
    UPDATE t_link
    SET enable_status = 1
    WHERE gid = #{gid}
      AND full_short_url = #{fullShortUrl}
      AND del_flag = 0
      AND enable_status = 0
      AND max_access_count IS NOT NULL
      AND current_access_count >= max_access_count
</update>
```

而在服务层，`tryConsumeAccessQuota` 对 SQL 影响行数的解释也非常明确：`updated > 0` 代表消费成功，`updated == 0` 代表额度耗尽或短链不可用，此时立即禁用并写空值缓存，避免重复回源。

```java
private boolean tryConsumeAccessQuota(String gid, String fullShortUrl) {
    int updated = baseMapper.incrementCurrentAccessCount(gid, fullShortUrl);
    if (updated > 0) {
        bizMetricsKit.increment("shortlink.biz.shortlink.quota.consume", "result", "success");
        return true;
    }
    bizMetricsKit.increment("shortlink.biz.shortlink.quota.consume", "result", "exhausted");
    baseMapper.disableByAccessLimit(gid, fullShortUrl);
    stringRedisTemplate.delete(String.format(GOTO_SHORT_LINK_KEY, fullShortUrl));
    stringRedisTemplate.opsForValue().set(String.format(GOTO_IS_NULL_SHORT_LINK_KEY, fullShortUrl), "-", 30, TimeUnit.MINUTES);
    return false;
}
```
