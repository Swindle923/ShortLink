# 短链接系统全链路深度原理文档（代码级）

> 本文基于当前仓库实现编写，目标是把“架构图 + 关键链路 + 数据模型 + 并发一致性 + 故障处理 + 可观测性”一次讲透。  
> 与 `01~06` 专题文档不同，本文强调端到端与代码落点，不只讲“做了什么”，更讲“为什么这样做”。

---

## 1. 代码基线与运行形态

### 1.1 模块结构

- `admin`：管理后台业务（用户、分组、短链管理、统计查询、回收站）。
- `project`：短链核心域（生成、跳转、统计入库、回收站落库、URL标题等）。
- `aggregation`：单体编排层，把 `admin + project` 组合为一个应用进程。
- `console-vue`：前端控制台。

### 1.2 三种入口

- 聚合入口：`aggregation/.../AggregationServiceApplication.java`
- 管理端入口：`admin/.../ShortLinkAdminApplication.java`
- 核心域入口：`project/.../ShortLinkApplication.java`

### 1.3 实际请求流（默认单体）

`console-vue -> admin/controller -> ShortLinkActualRemoteService -> LocalShortLinkActualRemoteService -> project/service -> mapper -> MySQL/Redis`

说明：  
`admin` 不直接依赖 `project controller`，而是经 `ShortLinkActualRemoteService` 抽象，单体模式下由 `aggregation` 的 `LocalShortLinkActualRemoteService` 做本地编排，保留未来拆分远程调用的演进空间。

## 2. 核心数据模型（为什么这样设计）

SQL 基线见：`resources/database/create-tables.sql`。

### 2.1 主实体

- `t_link`：短链主表（业务真相源）
  - 关键字段：
    - `full_short_url`：短链全地址（业务唯一标识核心）
    - `gid`：分组隔离键
    - `origin_url`：目标长链
    - `enable_status`：是否可用（0 启用 / 1 禁用）
    - `valid_date_type + valid_date`：有效期策略
    - `max_access_count + current_access_count`：访问配额模型
    - `del_flag + del_time`：软删语义
  - 关键索引：
    - 唯一键 `idx_unique_full-short-url (full_short_url, del_time)`  
      允许同一个短链“历史记录共存”（旧记录 `del_time != 0`，新记录 `del_time=0`），支持迁移分组时“软删旧记录 + 插入新记录”。

- `t_link_goto`：短链到分组映射（轻量路由表）
  - 唯一键 `idx_full_short_url(full_short_url)`  
  - 作用：跳转回源时先拿 `gid`，再回 `t_link` 查询；避免在某些写路径里直接全表搜主业务记录。

### 2.2 统计实体（按维度拆表）

- `t_link_access_stats`：按天/小时/星期聚合 PV/UV/UIP
- `t_link_locale_stats`：地域维度
- `t_link_os_stats`：操作系统维度
- `t_link_browser_stats`：浏览器维度
- `t_link_device_stats`：设备维度
- `t_link_network_stats`：网络维度
- `t_link_access_logs`：明细日志
- `t_link_stats_today`：今日聚合

设计动机：  
统计读场景是“多维聚合 + 图表”，把维度预聚合能显著降低查询时聚合成本，避免每次都从明细日志全量扫描。

## 3. 缓存键与数据生命周期

常量见：`project/.../RedisKeyConstant.java`

- `short-link:goto:%s`：短链跳转缓存（值=origin_url）
- `short-link:is-null:goto_%s`：空值缓存（防穿透）
- `short-link:lock:goto:%s`：跳转回源互斥锁
- `short-link:lock:update-gid:%s`：短链分组迁移读写锁（读锁给统计消费，写锁给迁组）
- `short-link:stats:uv:` / `short-link:stats:uip:`：去重集合键（UV/UIP）
- `short-link:stats-stream`：Redis Stream 统计消息流
- `short-link:stats-stream:only-group`：Stream 消费组

生命周期规则：

- 跳转缓存 TTL = 短链有效期剩余时间（通过 `LinkUtil.getLinkCacheValidTime` 计算）。
- 失效短链/已删除短链写入空值缓存（默认 30 分钟）避免 DB 被持续打穿。
- 修改短链目标或有效期时主动删 `goto` 缓存，必要时删 `is-null` 缓存，保证“后续访问重新回源拿新值”。

## 4. 创建链路（从请求到入库到缓存）

核心实现：`project/.../ShortLinkServiceImpl#createShortLink`。

### 4.1 时序

1. 白名单校验（演示模式防恶意跳转）  
2. 生成短码（`HashUtil.hashToBase62(origin+UUID)`，最多重试 10 次）  
3. 构建 `t_link` 与 `t_link_goto` 两个 DO  
4. 写数据库（`t_link` 后 `t_link_goto`）  
5. 写 Redis 跳转缓存（预热）  
6. 把 fullShortUrl 加入布隆过滤器  
7. 返回 `http://{domain}/{suffix}`

### 4.2 冲突与一致性策略

- 第一层：布隆过滤器提前判重，减少 DB 冲突概率。
- 第二层：数据库唯一键兜底，捕获 `DuplicateKeyException`。
- 第三层：可选锁版本 `createShortLinkByLock`（全局锁串行化生成）。

取舍：

- 无锁路径吞吐更高，依赖 Bloom + 唯一键兜底。
- 锁路径更稳但吞吐低，适用于极端冲突场景。

## 5. 跳转链路（高并发读路径核心）

核心实现：`project/.../ShortLinkServiceImpl#restoreUrl`。

### 5.1 快路径（缓存命中）

1. 组装 `fullShortUrl`
2. 读 `short-link:goto:%s`
3. 命中则先扣配额 `incrementCurrentAccessCount`
4. 发送统计消息
5. `302` 跳转长链

### 5.2 慢路径（缓存未命中）

1. Bloom 不包含 -> 直接 `notfound`
2. 命中空值缓存 -> `notfound`
3. 获取 `LOCK_GOTO_SHORT_LINK_KEY`（防回源并发风暴）
4. 双检缓存（防止锁等待期间其他线程已回填）
5. 查 `t_link_goto` 取 `gid`
6. 按 `gid + full_short_url + 状态` 查 `t_link`
7. 校验有效期
8. 扣配额（失败则禁用并写空值缓存）
9. 回填 `goto` 缓存
10. 发送统计消息并跳转

### 5.3 访问配额模型

SQL 在 `LinkMapper.xml`：

- `incrementCurrentAccessCount`：  
  `current_access_count < max_access_count` 才允许 +1（单 SQL 原子扣减）。
- 返回 0 表示额度耗尽或链接不可用。
- 耗尽时执行 `disableByAccessLimit`，并删除 `goto` 缓存、写空值缓存，后续直接 notfound。

这避免了“先查后改”带来的并发超扣问题。

## 6. 统计链路（异步化 + 多维落库）

### 6.1 生产端

位置：`ShortLinkServiceImpl#shortLinkStats` + `ShortLinkStatsSaveProducer#send`

- 跳转请求线程只负责封装 `ShortLinkStatsRecordDTO` 并 `XADD` 到 Redis Stream。
- 用户请求不等待统计落库，核心目标是降低跳转延迟。

### 6.2 消费端

位置：`RedisStreamConfiguration` + `ShortLinkStatsSaveConsumer`

- 消费组：`short-link:stats-stream:only-group`
- 消费者：`stats-consumer`
- 单线程消费线程池（1 core/1 max），保证消费顺序与实现简单性。

### 6.3 幂等与失败恢复

`ShortLinkStatsSaveConsumer#onMessage` 中：

- 使用 `MessageQueueIdempotentHandler` 判断消息是否正在消费/已完成。
- 处理失败时删除“处理中标记”，抛出异常触发重试。
- 成功后删除 stream 记录并标记完成。

### 6.4 落库内容

`actualSaveShortLinkStats` 一次消费会更新：

- `t_link_access_stats`（日/小时/星期）
- `t_link_locale_stats`（调用高德 IP 归属）
- `t_link_os_stats`
- `t_link_browser_stats`
- `t_link_device_stats`
- `t_link_network_stats`
- `t_link_access_logs`（明细）
- `t_link` 的累计总量 `total_pv/uv/uip`
- `t_link_stats_today`

### 6.5 与分组迁移并发的协调

- 统计消费使用 `LOCK_GID_UPDATE_KEY` 的**读锁**。
- 短链迁移分组（update 时 gid 变化）使用同一把锁的**写锁**。

这样保证“迁移过程中统计写入不会拿错 gid 或写到不一致状态”。

## 7. 更新链路与分组迁移语义

核心实现：`ShortLinkServiceImpl#updateShortLink`。

### 7.1 不变更 gid（同组更新）

- 直接 `UPDATE t_link` 修改目标地址/描述/有效期/配额。
- 若关键字段变化（origin/有效期）则删缓存，触发下次访问重建。

### 7.2 变更 gid（跨组迁移）

在写锁保护下：

1. 把旧记录置为软删（`del_flag=1, del_time=now`）
2. 插入新记录（新 gid，其他业务字段继承）
3. 更新 `t_link_goto` 的 gid 映射

配合唯一键 `(full_short_url, del_time)`，可保留历史又保证当前有效记录唯一。

## 8. 分组、用户、认证与风控

### 8.1 用户登录态

`admin/.../UserServiceImpl`：

- 登录后写 Redis Hash：`USER_LOGIN_KEY + username`，field=token，value=用户 JSON。
- TTL 30 分钟；重复登录复用已有 token 并续期。

### 8.2 过滤器链顺序（admin）

`UserConfiguration`：

1. `UserTransmitFilter`（order=0）：从 header 透传用户上下文到 `ThreadLocal`
2. `UserAuthenticationFilter`（order=5）：校验 token，保护 `/api/short-link/admin/v1/**`
3. `UserFlowRiskControlFilter`（order=10，可配置开关）：Lua 限流（时间窗 + 最大访问）

### 8.3 角色模型

- `t_user.role` 支持 `ADMIN/USER`
- 管理员接口（当前仅 `/api/short-link/admin/v1/user/role`）要求 `ADMIN`

### 8.4 分组创建

`GroupServiceImpl#saveGroup`：

- 按用户维度加锁（`LOCK_GROUP_CREATE_KEY + username`）
- 校验分组上限（默认 20）
- 先写 `t_group_unique`（唯一 gid）再写 `t_group`
- gid 同步写入 Bloom，降低后续冲突

## 9. 回收站语义

核心实现：`project/.../RecycleBinServiceImpl`

- 放入回收站：`enable_status=1` + 删跳转缓存（不可访问）
- 恢复：`enable_status=0` + 删空值缓存（可重新访问）
- 彻底移除：软删（`del_flag=1, del_time=now`）

admin 侧回收站分页先查当前用户全部 gid，再调用 project 聚合查询，确保用户只能看自己分组范围数据。

## 10. 统计查询模型（图表接口）

核心实现：`ShortLinkStatsServiceImpl`

特征：

- 先做“分组归属校验”再查统计（防越权）。
- 对时间范围按天补齐零值，保证前端图表连续。
- 输出维度包括：
  - 日趋势、小时分布、周分布
  - 地域分布、OS、浏览器、设备、网络
  - 高频 IP、新老访客占比
  - 明细访问记录分页

## 12. 当前实现的关键优势与边界

### 12.1 优势

- 跳转链路强缓存化，命中路径极短。
- 回源链路有 Bloom + 空值缓存 + 分布式锁三重防护。
- 统计异步化，主链路低延迟。
- 配额扣减用单 SQL 原子条件更新，天然抗并发超卖。
- 分组迁移与统计写入通过读写锁做跨链路一致性保护。

### 12.2 边界与可演进点

- Redis Stream 消费当前单线程，极高流量下可能成为统计吞吐瓶颈。
- 地域解析依赖外部高德 API，外部抖动会影响统计维度完整性。
- 统计写路径是“多表逐步写”，若要更高吞吐可考虑批量聚合后异步落库。
- Sentinel 规则目前仅对创建接口做极低阈值（QPS=1）示例化配置，生产需动态治理。
