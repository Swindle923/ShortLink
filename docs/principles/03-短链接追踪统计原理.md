# 短链接追踪统计原理（论文叙事版）

短链接系统里最容易被忽略、但实际上最能体现工程质量的部分并不是“生成一个短码”，而是“如何在不拖慢跳转链路的前提下，把统计做准、做全、做稳”。如果把跳转系统看成一条高速公路，那么统计系统就是沿路布设的传感网络：它必须尽可能少地干扰主车道，又要持续、可靠地采集流量、访客和环境信息。这个项目在实现上选择了非常典型而且有效的路线：跳转主链路只负责采集和投递，真正的多维统计计算全部放到异步消费者里执行。对应代码上，主流程在 `ShortLinkServiceImpl#shortLinkStats`，投递动作由 `ShortLinkStatsSaveProducer#send` 完成，消息介质是 Redis Stream，主题键是 `SHORT_LINK_STATS_STREAM_TOPIC_KEY`。

从系统行为上看，这个决策的意义非常直接。用户发起短链访问后，`restoreUrl` 的目标是尽快返回 302，而不是在请求线程内做大量数据库写入。于是主线程只做轻量动作：拼装 `ShortLinkStatsRecordDTO`，把当前请求上下文（如 UV、IP、浏览器、系统、设备、网络类型和时间戳）序列化为消息体，然后写入 Stream。至此，跳转请求可以结束。换句话说，跳转是否“快”与统计是否“慢”被结构性解耦了，这种解耦是整套统计设计成立的基础。

消息进入 Stream 后，消费体系由 `RedisStreamConfiguration` 和 `ShortLinkStatsStreamInitializeTask` 负责启动。前者配置了消费者订阅参数，后者确保消费组存在。当前实现采用单消费者线程模型，执行上相对保守，但好处是处理顺序清晰、状态更容易解释，也降低了多并发消费者同时写同一维度统计记录时的竞争复杂度。真正的数据落库逻辑在 `ShortLinkStatsSaveConsumer#actualSaveShortLinkStats`，这段代码不是一次“写一张表”，而是一组有内在约束关系的统计动作：它先通过 `fullShortUrl` 找到 `gid`，接着围绕同一个访问事件更新访问趋势、地域分布、设备结构、网络结构和访问日志，并同步推进主表累计值和今日统计值。

这里有一个容易被论文忽略、但在工程里非常关键的问题：消息系统天然会面对重复投递、消费者重启、处理超时等情况，如果没有幂等控制，统计数据很快就会漂移。当前项目在 `onMessage` 中引入了 `MessageQueueIdempotentHandler`，将“是否在处理”“是否已完成”这两个状态显式化。这样做的效果是，即使同一条消息被再次拉取，只要被判定为“已完成”就会快速跳过；如果处于“处理中但未完成”的中间态，则抛出异常等待重试，避免产生半提交统计。这种处理方式虽然没有引入复杂事务消息框架，但已经形成了足够实用的“最终一致 + 幂等防重”闭环。

统计口径本身也不是写几个 `count(*)` 那么简单。PV 在该系统中是访问次数，每次有效跳转都应记一次；UV 与 UIP 则需要去重语义，且去重状态不能放在本地内存，否则多实例部署会失真。因此系统在跳转阶段先用 Redis Set 做去重判定，再把 `uvFirstFlag` 和 `uipFirstFlag` 随消息带到消费端。消费端据此决定本次事件是否推动 UV/UIP 递增。这种“采集端判定 + 消费端落账”的分工，是统计精度与吞吐的折中点：它避免消费端重复计算请求上下文，同时保证口径统一。

再往深一层看，统计系统并不是孤立存在的。短链支持分组迁移，意味着同一个 `fullShortUrl` 在不同时间可能对应不同 `gid`。如果迁移和统计消费并发发生，统计就可能写到错误分组。为了解决这个跨链路一致性问题，代码在统计消费侧使用 `LOCK_GID_UPDATE_KEY` 的读锁，而在短链迁组更新中使用同一把锁的写锁。这个设计很像数据库里的读写互斥协议：统计可以并发读，但只要有迁组写入，读就会被协调，让“访问归属”在时间线上始终可解释。

查询层则是另一个维度的工程问题。`ShortLinkStatsServiceImpl` 并不是简单把数据库结果原样返回，而是做了大量“面向可视化”的二次加工，比如日期补零、比例计算、新老访客分类、Top IP 排序、按小时与按周分布重建。这样做的结果是后端承担更多计算，前端拿到更直接可渲染的数据模型。对于毕业设计场景，这一点尤其重要，因为它体现了你对“数据采集、数据存储、数据服务”三层职责边界的完整理解。

如果从论文评价角度总结这套实现，它的亮点不在于用了某个“高级中间件”，而在于一组正确的工程选择叠加在一起：跳转线程不做重活，统计通过异步落地；消息处理显式幂等，避免重复记账；口径计算前置并标准化，保证跨维度一致；迁组与统计通过锁协同，维持归属正确性。这些机制共同构成了一个可以长期运行、能解释异常行为、并且具备扩展潜力的统计子系统。

如果把这套机制放回具体代码，首先可以看到生产端几乎是最小开销实现：只有一次 `XADD`，没有任何同步落库动作。

```java
@Component
@RequiredArgsConstructor
public class ShortLinkStatsSaveProducer {
    private final StringRedisTemplate stringRedisTemplate;
    public void send(Map<String, String> producerMap) {
        stringRedisTemplate.opsForStream().add(SHORT_LINK_STATS_STREAM_TOPIC_KEY, producerMap);
    }
}
```

消费端则承担了可靠性职责。`onMessage` 的实现把“处理中”“已完成”两个状态都显式化，避免重复消费导致的重复记账。注意异常分支里会清理处理状态并抛出异常，允许后续重试继续推进。

```java
@Override
public void onMessage(MapRecord<String, String, String> message) {
    String stream = message.getStream();
    RecordId id = message.getId();
    if (messageQueueIdempotentHandler.isMessageBeingConsumed(id.toString())) {
        if (messageQueueIdempotentHandler.isAccomplish(id.toString())) {
            return;
        }
        throw new ServiceException("消息未完成流程，需要消息队列重试");
    }
    try {
        Map<String, String> producerMap = message.getValue();
        ShortLinkStatsRecordDTO statsRecord = JSON.parseObject(producerMap.get("statsRecord"), ShortLinkStatsRecordDTO.class);
        actualSaveShortLinkStats(statsRecord);
        stringRedisTemplate.opsForStream().delete(Objects.requireNonNull(stream), id.getValue());
    } catch (Throwable ex) {
        messageQueueIdempotentHandler.delMessageProcessed(id.toString());
        throw ex;
    }
    messageQueueIdempotentHandler.setAccomplish(id.toString());
}
```

`actualSaveShortLinkStats` 里真正体现了“多维统计一次消费全部推进”的思路。这里先拿读锁，再查 `t_link_goto` 获取当前 gid，然后同时更新趋势表、维度表、明细日志和主表累计值。只要读完这段代码，就能清楚为什么这套统计是“写放大但读友好”的架构。

```java
public void actualSaveShortLinkStats(ShortLinkStatsRecordDTO statsRecord) {
    String fullShortUrl = statsRecord.getFullShortUrl();
    RReadWriteLock readWriteLock = redissonClient.getReadWriteLock(String.format(LOCK_GID_UPDATE_KEY, fullShortUrl));
    RLock rLock = readWriteLock.readLock();
    rLock.lock();
    try {
        ShortLinkGotoDO shortLinkGotoDO = shortLinkGotoMapper.selectOne(
                Wrappers.lambdaQuery(ShortLinkGotoDO.class).eq(ShortLinkGotoDO::getFullShortUrl, fullShortUrl)
        );
        String gid = shortLinkGotoDO.getGid();
        linkAccessStatsMapper.shortLinkStats(...);
        linkLocaleStatsMapper.shortLinkLocaleState(...);
        linkOsStatsMapper.shortLinkOsState(...);
        linkBrowserStatsMapper.shortLinkBrowserState(...);
        linkDeviceStatsMapper.shortLinkDeviceState(...);
        linkNetworkStatsMapper.shortLinkNetworkState(...);
        linkAccessLogsMapper.insert(...);
        shortLinkMapper.incrementStats(gid, fullShortUrl, 1, statsRecord.getUvFirstFlag() ? 1 : 0, statsRecord.getUipFirstFlag() ? 1 : 0);
        linkStatsTodayMapper.shortLinkTodayState(...);
    } finally {
        rLock.unlock();
    }
}
```

最后补一段配置细节。Redis Stream 消费并不是“自动发生”的，`RedisStreamConfiguration` 明确了消费线程数、批量拉取、消费组与消费者名，这些参数直接决定吞吐与堆积行为。

```java
StreamMessageListenerContainerOptions<String, MapRecord<String, String, String>> options =

    .build();
```
