# 监控系统设计与实现（论文版）

## 1. 功能介绍
监控系统模块用于构建短链接平台的可观测能力闭环，覆盖指标采集、日志检索、可视化看板和告警通知。其目标是将“系统是否健康”从主观感受转化为可量化证据。

模块组成如下：
- 指标采集：Spring Boot Actuator + Micrometer
- 指标存储与规则：Prometheus
- 告警路由：Alertmanager
- 可视化：Grafana
- 日志聚合：Promtail + Loki

## 2. 流程介绍
监控主流程可分为四段：
1. 应用暴露 `/actuator/prometheus` 指标端点。
2. Prometheus 周期抓取指标并执行告警规则。
3. Grafana 拉取 Prometheus/Loki 数据渲染看板与日志查询。
4. 告警由 Alertmanager 聚合后分发到通知渠道。

## 3. 具体原理
### 3.1 指标原理
- 使用 AOP 对服务层方法做自动埋点，输出调用次数与延迟分位。
- 通过业务埋点工具输出语义指标（创建成功率、配额耗尽等）。
- 技术指标与业务指标并行，形成多维度诊断基础。

### 3.2 日志原理
- Promtail 采集容器日志并附加标签后推送 Loki。
- Grafana 通过 LogQL 做跨实例统一检索。
- 指标异常可快速定位到具体日志上下文，缩短排障路径。

### 3.3 告警原理
- 告警规则在 Prometheus 侧表达式化定义，支持版本化治理。
- Alertmanager 对告警做分组、抑制和路由，避免告警风暴。
- 告警闭环与业务看板联动，形成可执行的运维流程。

## 4. 设计思路
监控系统采用“分层观测 + 业务优先”的设计策略：
- 分层观测：应用指标、业务指标、日志证据三层联动。
- 业务优先：不仅关注 CPU/内存，还关注业务健康状态变化。
- 规则先行：告警基于规则持续评估，而非人工看图判断。
- 可演进：当前架构可平滑扩展链路追踪与自动化 Runbook。

该模块证明了系统不仅“能跑”，还能在异常时“可定位、可解释、可处置”，这也是毕业设计中工程能力的重要体现。
